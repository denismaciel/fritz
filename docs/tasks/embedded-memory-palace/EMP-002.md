# EMP-002: SQLite runtime + extension loading strategy

**Type:** Task

## Goal

Prove and codify how the first stack will use SQLite, FTS5, and `sqlite-vec` in this repo and in a later extracted package.

## Places To Change

- `pkg/memorypalace/sqlite/`
- `flake.nix`
- `package.nix`
- `go.mod`
- `docs/memory-palace-storage-notes.md`

## Scope

- choose the SQLite driver approach for the package
- document and implement how FTS5 is enabled
- document and implement how `sqlite-vec` is loaded or linked
- add a tiny runtime probe/helper that reports capability availability without coupling it to Fritz CLI
- add tests that make capability failure explicit

## Non-goals

- canonical schema
- search implementation
- Ollama integration
- Fritz-level commands or UI

## Design Notes

- this ticket is meant to burn down packaging risk early
- extension loading strategy must work in local dev and CI, or fail clearly with actionable errors
- keep the capability probe under the SQLite adapter, not the core package
- if `sqlite-vec` is not immediately shippable in this env, keep a clean fallback path for FTS-only progress

## Acceptance Criteria

- the repo has one documented SQLite/FTS5/`sqlite-vec` strategy
- tests can detect missing vector capability cleanly
- later tickets can depend on a stable adapter seam instead of ad hoc setup

## Dependencies

- `EMP-001`

## Validation

- `go test ./pkg/memorypalace/sqlite/...`
- `rg -n "sqlite-vec|FTS5|capabilit" pkg/memorypalace/sqlite docs/memory-palace-storage-notes.md`
