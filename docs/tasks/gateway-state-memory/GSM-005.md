# GSM-005: durable memory filesystem layout + loader

**Type:** Task

## Goal

Add durable memory as explicit files, separate from ops state, with a small loader seam.

## Places To Change

- `internal/memory/`
- `internal/prompt/`
- `internal/config/`

## Scope

- define durable memory roots and file names
- support:
  - `MEMORY.md`
  - `memory/*.md`
- add loader that reads memory docs deterministically
- keep output typed enough for prompt assembly later
- add tests for file discovery/order/loading

## Non-goals

- automatic memory extraction
- retrieval index
- writing memory from the agent loop

## Design Notes

- durable memory is not gateway ops state
- keep it human-editable and repo-local first
- use OpenClaw concept docs as shape refs:
  - `/home/denis/tmp/openclaw-EUwHTl/docs/concepts/memory-builtin.md`
  - `/home/denis/tmp/openclaw-EUwHTl/docs/reference/session-management-compaction.md`
- deterministic load order matters; keep it simple and documented

## Acceptance Criteria

- memory loader can read `MEMORY.md` and `memory/*.md`
- tests prove deterministic ordering and missing-file behavior
- no mutable JSON ops-state code depends on memory loader

## Dependencies

- `GSM-004`

## Validation

- `go test ./internal/memory ./internal/prompt`
- `rg -n "MEMORY.md|memory/\\*\\.md" internal`
