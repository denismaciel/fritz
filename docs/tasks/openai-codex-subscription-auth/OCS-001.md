# OCS-001: provider abstraction + config surface

**Type:** Task

## Goal

Add smallest provider abstraction needed so `fritz` can support both Gemini and `openai-codex` without hardcoding Gemini in app bootstrap.

## Places To Change

- `internal/config/`
- `internal/app/`
- `internal/command/`
- new small `internal/provider/` package if needed

## Scope

- add provider selection to runtime config and CLI flags
- keep Gemini as default provider
- replace hardcoded Gemini gateway creation with a provider switch seam
- expose provider details in `doctor`
- add focused config/command tests
- keep `internal/model.Gateway` unchanged

## Non-goals

- auth storage
- OAuth login flow
- Codex HTTP transport
- large dynamic model registry work
- changing model request/response contracts

## Design Notes

- prefer a small explicit provider enum/string over a generalized plugin system
- keep existing Gemini path behavior unchanged when provider is unset
- provider choice should be validated early and fail cleanly
- app owns provider selection; transport impls stay behind existing gateway seam

## Acceptance Criteria

- runtime config can express `gemini` vs `openai-codex`
- default behavior stays Gemini-compatible
- app bootstrap no longer assumes Gemini by construction
- config and command parsing tests cover provider selection
- `internal/model.Gateway` interface is unchanged

## Dependencies

- `None.`

## Validation

- `go test ./internal/config ./internal/command ./internal/app`
- `rg -n "openai-codex|provider" internal/config internal/command internal/app`
