# PTP-001 path resolution parity for file tools

## Goal

Port Pi path-resolution behavior used by file tools. Get path expansion and macOS filename normalization right before deeper read/edit parity work.

## Places To Change

- `internal/tool/tool.go`
- `internal/tool/tool_test.go`
- new helper file under `internal/tool/` if needed

## Scope

- add path expansion helpers for `~` and `~/...`
- resolve relative vs absolute paths consistently
- handle Unicode/path variants Pi already tests
- wire helpers into file tools

## Non-goals

- tool output formatting parity
- image handling
- mutation queue
- Windows-specific shell behavior

## Design Notes

- keep path logic pure where possible
- prefer one shared resolver for `read`/`write`/`edit`/future `grep`/`find`/`ls`
- stay rooted in cwd policy; do not weaken sandboxing

## Acceptance Criteria

- local Go tests cover each checklist item below
- file tools use shared resolver logic
- `go test ./...` stays green

## Dependencies

- none

## Validation

- `go test ./internal/tool ./...`

## Test Checklist

- [x] should expand ~ to home directory
- [x] should expand ~/path to home directory
- [x] should normalize Unicode spaces
- [x] should resolve absolute paths as-is
- [x] should resolve relative paths against cwd
- [x] should resolve existing file path
- [x] should handle NFC vs NFD Unicode normalization (macOS filenames with accents)
- [x] should handle curly quotes vs straight quotes (macOS filenames)
