# PTP-002 read tool parity

## Goal

Port Pi read-tool behavior, incl truncation, offsets, limits, and image/text detection.

## Places To Change

- `internal/tool/tool.go`
- `internal/tool/tool_test.go`
- `internal/model/model.go`
- `internal/app/app.go`
- any new content-block types under `internal/`

## Scope

- line/byte truncation
- offset + limit semantics
- truncation metadata
- image sniffing by file magic, not extension
- return shape rich enough for text + image payloads if needed

## Non-goals

- image filtering policy at prompt-conversion layer
- write/edit/bash changes
- grep/find/ls

## Design Notes

- current string-only tool result may be too small; widen only as needed
- keep model-facing tool results serializable and provider-agnostic
- preserve cwd sandbox

## Acceptance Criteria

- local tests cover each checklist item below
- read tool supports offset/limit/truncation behavior
- image vs text behavior matches Pi intent
- `go test ./...` stays green

## Dependencies

- `PTP-001`

## Validation

- `go test ./internal/tool ./internal/app ./...`

## Test Checklist

- [x] should read file contents that fit within limits
- [x] should handle non-existent files
- [x] should truncate files exceeding line limit
- [x] should truncate when byte limit exceeded
- [x] should handle offset parameter
- [x] should handle limit parameter
- [x] should handle offset + limit together
- [x] should show error when offset is beyond file length
- [x] should include truncation details when truncated
- [x] should detect image MIME type from file magic (not extension)
- [x] should treat files with image extension but non-image content as text
- [x] should always read images (filtering happens at convertToLlm layer)
- [x] should read text files normally
