# PTP-007 bash parity

## Goal

Port Pi bash-tool behavior plus the small agent-loop integration Pi already tests.

## Places To Change

- `internal/tool/tool.go`
- `internal/tool/tool_test.go`
- `internal/app/app.go`
- `internal/app/app_test.go`
- `internal/chat/core.go`
- `internal/model/model.go` if needed

## Scope

- command execution
- output/error capture
- timeout/cancel behavior
- command prefix support if kept
- reusable local bash ops seam
- bash output fed back into model context
- Windows close-handling regression if feasible in Go

## Non-goals

- interactive shell sessions
- TUI bash rendering
- package manager/install commands

## Design Notes

- keep bash backend behind a narrow seam
- preserve current safety rails unless Pi behavior requires more
- if Windows-specific regression is hard locally, keep test names tracked and note blocker

## Acceptance Criteria

- local tests cover checklist items below, or ticket notes record concrete blocker
- bash output can be surfaced back into agent context
- `go test ./...` stays green

## Dependencies

- none

## Validation

- `go test ./internal/tool ./internal/app ./...`

- [x] should execute simple commands
- [x] should handle command errors
- [x] should respect timeout
- [x] should throw error when cwd does not exist
- [x] should handle process spawn errors
- [x] should prepend command prefix when configured
- [x] should include output from both prefix and command
- [x] should work without command prefix

- [x] should preserve executeBash sanitization when using local bash operations
- [x] executeBash resolves after the shell exits even if inherited stdio handles stay open
- [x] bash tool resolves after the shell exits even if inherited stdio handles stay open
