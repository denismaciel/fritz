# PIS-001 session file model and storage layout

## Goal

Define our session JSONL format and storage layout, based on Pi's proven structure, but trimmed to what we need first.

## Scope

- session header type
- entry base with `id` + `parentId`
- message entry type
- initial session file naming and directory layout
- leave compaction/branch summary entry payloads stubbed if needed

## Non-goals

- persistence logic
- resume logic
- compaction behavior

## Pi reference files

- [`/tmp/pi-mono/packages/coding-agent/docs/session.md`](/tmp/pi-mono/packages/coding-agent/docs/session.md)
- [`/tmp/pi-mono/packages/coding-agent/src/core/session-manager.ts`](/tmp/pi-mono/packages/coding-agent/src/core/session-manager.ts)
  - `SessionHeader`
  - `SessionEntryBase`
  - entry type defs near top
  - `CURRENT_SESSION_VERSION`

## Notes

- Pi uses one append-only JSONL file with tree entries
- Pi stores sessions under cwd-derived dirs
- we can start with fewer entry kinds, but shape should not paint us into a corner

## Validation

- format/unit tests for parsing + serializing session header and entries
