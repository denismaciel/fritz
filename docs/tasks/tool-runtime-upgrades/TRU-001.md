# TRU-001 structured tool result payloads

## Goal

Replace string-biased tool results with typed content parts and typed detail payloads, while preserving current visible behavior.

## Places To Change

- `internal/tool/types.go`
- `internal/model/model.go`
- `internal/gemini/client.go`
- `internal/chat/core.go`
- `internal/app/app.go`
- tool tests under `internal/tool/`

## Scope

- define typed tool content parts
- define typed per-tool detail structs where needed
- replace `Details map[string]any` in core result types
- keep current tool names and current user-visible output text stable

## Non-goals

- adding a second provider
- UI rendering layer
- session persistence
- discovery behavior changes

## Design Notes

- favor a small content model: text, image, maybe future binary/file ref
- tool result payload should be provider-agnostic
- Gemini adapter should map from core typed payloads, not invent them

## Acceptance Criteria

- core tool result types no longer rely on generic detail maps
- `read` image output no longer tunnels image data only through ad hoc maps
- existing tests pass with updated assertions
- parity validator still passes

## Dependencies

- none

## Validation

- `go test ./...`
- `uv run scripts/validate_pi_tool_test_parity.py`
