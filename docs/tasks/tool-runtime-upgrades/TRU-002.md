# TRU-002 provider-agnostic model/tool boundary

## Goal

Make the model boundary speak typed, provider-agnostic tool messages so future provider or transport swaps stay local to adapters.

## Places To Change

- `internal/model/model.go`
- `internal/gemini/client.go`
- `internal/chat/core.go`
- `internal/app/app.go`
- tests in `internal/model/`, `internal/gemini/`, `internal/app/`

## Scope

- refine function-call / function-response message types
- ensure tool request/response payloads are typed and provider-neutral
- reduce Gemini-specific JSON assumptions in core

## Non-goals

- adding new providers
- changing tool behavior
- bash runtime changes

## Design Notes

- core should know about agent messages and tool calls/results, not Gemini wire fields
- adapters should be the only place that care about `functionCall`, `functionResponse`, etc
- keep the boundary compatible with structured tool results from `TRU-001`

## Acceptance Criteria

- core model types are provider-agnostic
- Gemini adapter contains the provider-specific translation
- current tool loop behavior stays green
- parity validator still passes

## Dependencies

- `TRU-001`

## Validation

- `go test ./...`
- `uv run scripts/validate_pi_tool_test_parity.py`
